## 잘 된 것들

### 1. 아이디어와 포지셔닝

"프로젝트 폴더 안에 사는 API 클라이언트"라는 컨셉 자체가 명확하다. Postman은 요청 기록을 클라우드에 가두고, curl은 기억하기 어렵다. 이 사이 어딘가를 잘 파고들었다.

- 단일 파일(`server.mjs`)로 어느 프로젝트에나 붙여 넣기 가능
- 스냅샷/템플릿을 파일로 저장해 Git 추적 가능
- 커스텀 함수로 토큰 자동 갱신 같은 반복 작업 자동화

### 2. 기술 선택

Hono + Hono JSX + HTMX + Alpine.js 조합은 이 프로젝트에 잘 맞다.

- **Hono**: 가볍고 번들에 유리
- **HTMX**: 부분 업데이트(사이드바, 스냅샷 목록)를 SPA 없이 처리
- **Alpine.js**: 폼 상태 관리를 별도 프레임워크 없이 해결
- **esbuild**: 빌드 속도 빠르고 단일 파일 번들 출력

프로덕션 빌드 시 HTMX, Alpine.js까지 인라인으로 번들링해 외부 CDN 없이 동작하는 것도 좋다.

### 3. 설정 구조

`config/default.json` + `POCKET_ENV=dev`로 환경별 오버라이드하는 계층 구조가 실용적이다. 프로젝트마다 `.env`처럼 `config/` 폴더를 Git에 넣으면 팀원 간 API 환경 공유가 된다.

### 4. README

기능 설명, 설정 옵션, 커스텀 함수 사용법까지 잘 정리되어 있다. 처음 보는 사람도 15분 안에 쓸 수 있을 것 같다.

## 문제점들

### [심각] RequestForm.tsx - 660줄짜리 괴물

가장 큰 문제. 핵심 컴포넌트인데 관리가 안 되는 수준에 가까워지고 있다.

**문제 1: Alpine.js 로직이 JSX 문자열 안에**

```tsx
// 이런 식으로 x-data 안에 300줄짜리 객체가 들어있음
<div x-data={`{
  method: 'GET',
  rawUrl: '',
  // ... 300줄 더 ...
}`}>
```

- IDE 자동완성 없음
- TypeScript 타입 체크 없음
- 테스트 불가
- 리팩토링하다가 따옴표 하나 실수하면 전체가 죽음

**문제 2: 정규식으로 함수 이름 추출**

```tsx
// 커스텀 함수 파일에서 함수 이름을 정규식으로 뽑음
matchAll(/(?:async\s+)?function\s+([a-zA-Z_$][\w$]*)/g)
```

화살표 함수(`const foo = () => {}`)는 인식 못 한다. 예제 코드(`auth.js`)가 마침 일반 함수 선언 방식이라 동작하지만, 다른 스타일로 작성하면 무음 실패.

**문제 3: 무음 실패 4군데**

```tsx
catch (e) {}                           // 커스텀 함수 실행 실패 - 아무 피드백 없음
catch (e) { /* ignore */ }             // JSON 파싱 실패 - 아무 피드백 없음
catch (e) { /* silently fail */ }      // URL 파라미터 파싱 실패
catch (e) { /* silently fail */ }      // 코드 생성 실패
```

사용자 입장에서 왜 안 되는지 알 수 없다.

**문제 4: URL의 슬래시 처리 버그 의심**

```tsx
.replace(/\/+$/, '')  // 이건 맞는데
.replace(/\\+$/, '')  // 이건 백슬래시를 지우는 코드 - URL에서 이게 왜?
```

이스케이프 처리가 혼재되어 있어 실제 버그로 이어질 수 있다.

### [심각] 에러 처리 전반

`request.tsx`에서 타입 캐스팅을 검증 없이 한다.

```tsx
let rawUrl = body.url as string;    // undefined일 수 있음
const method = body.method as string;  // undefined일 수 있음
```

요청 바디가 예상과 다른 형태면 런타임에서 터진다. TypeScript를 쓰는데 `as string`으로 막아두면 타입 체커를 속이는 셈이다.

---

### [보통] 보안: 커스텀 함수 실행

`functions/` 폴더의 `.js` 파일을 로드해 브라우저에서 `eval()` 수준으로 실행한다. 로컬 전용 도구이므로 실제 위협은 낮지만:

- `functions/` 폴더가 다른 프로세스에 의해 쓰일 수 있는 환경이라면 임의 코드 실행
- 생성된 fetch 코드의 따옴표 이스케이프가 불완전해 엣지 케이스에서 깨진 코드 생성 가능

로컬 개발 도구 수준에서는 감수할 수 있는 수준이지만, 문서에 명시는 해야 한다.

---

### [보통] 성능: 동기 파일 I/O

자동완성 제안을 요청할 때마다 스냅샷 파일 **전체**를 동기로 읽는다.

```ts
// snapshot.ts
const files = readdirSync(snapshotsDir);
files.forEach(file => {
  const data = readFileSync(...);  // 매번 전부 읽음
  // ...
});
```

스냅샷이 수백 개 넘어가면 체감될 수 있다. 캐싱이나 메타데이터 인덱스가 없다.

또한 스냅샷 삭제 기능은 있지만 **보존 정책이 없다**. 계속 쓰면 폴더가 무한정 커진다.

---

### [낮음] 오타 및 일관성

- 폴더명, 컴포넌트명이 `snapshorts`로 통일되어 있는데 이게 의도인지 오타인지 불명확. `snapshots`가 맞는 영어.
- `lang="ko"` 하드코딩 - 영어 환경 사용자는 어색할 수 있음

## 실사용 관점 시나리오

### 쓰기 좋은 상황

- **1인 개발자**가 자기 프로젝트 API를 빠르게 테스트할 때
- **팀 프로젝트**에서 API 테스트 케이스를 Git으로 공유하고 싶을 때 (templates 폴더 커밋)
- **인증 토큰 갱신**이 귀찮을 때 (커스텀 함수로 자동화)
- Postman 쓰기는 너무 무거운데 curl은 기억하기 싫을 때

### 불편한 상황

- 요청 수백 개를 관리해야 하는 **대규모 API 프로젝트**
- 팀원 여러 명이 동시에 다른 요청을 보내는 **협업 환경** (파일 충돌 가능)
- **CI/CD 파이프라인** 연동이 필요한 경우
- 화살표 함수 스타일로 커스텀 함수를 작성하는 경우 (인식 안 됨)
